本页作为开场页，展示主题“智能推荐系统”、团队归属“西安交通大学 · 智能推荐系统项目组”以及时间“2026 年 6 月”。整体作用是建立项目名称和汇报身份。

讲解时可用一句话点明项目定位：本项目围绕智能推荐系统展开，重点介绍系统目标、算法实现、前后端架构、后端多 Agent 与 RAG 能力，以及最终成果。

## 第 2 页：目录

本页展示汇报结构，共 7 个部分：团队分工介绍、项目背景与目标、系统架构设计、核心算法实现、前端功能模块与开发、后端架构设计、项目成果展示。

讲解时应强调汇报逻辑是从“为什么做”到“怎么设计”，再到“如何实现”和“最终效果”。其中后端架构设计是重点部分，包含技术栈、分层架构、多 Agent、RAG、记忆架构、API 与数据库。

## 第 3 页：团队分工介绍章节页

本页进入第一章，主题是团队分工。页面强调通过明确职责边界、高效协同和互补技能组合，构建稳定的协作体系。

讲解时可作为过渡页，说明项目由算法、后端、前端、数据、部署等方向协同完成，各成员既承担算法任务，也参与工程落地。

## 第 4 页：团队分工介绍

本页说明团队分工理念：每位成员约 70% 参与 AI 算法，30% 参与工程技能建设，形成“算法 + 工程”双驱动梯队。具体分工包括统筹与架构、算法与后端、前端与可视化、数据工程与增强、部署与工程优化。

讲解重点是突出团队不是孤立开发，而是围绕完整项目链路分工协作：有人负责架构和进度，有人负责模型训练与 FastAPI 服务，有人负责 Vue 前端和 ECharts 可视化，有人负责数据清洗和增强，有人负责 Docker 部署、推理加速与工程优化。

## 第 5 页：项目背景与目标章节页

本页进入第二章，强调通过系统化规划构建可持续价值闭环。页面将项目分析分成两部分：深度洞察需求原点，以及目标锚定、驱动价值落地。

讲解时可说明本章回答两个问题：传统推荐系统为什么需要升级，以及本项目希望达成什么样的功能和性能目标。

## 第 6 页：项目背景：传统推荐系统面临多重挑战

本页列出传统推荐系统的四类痛点：信息过载与效率低下、冷启动难题、数据孤岛、可解释性与信任不足。页面最后指出行业需要智能化、自动化、可解释的智能分析体系。

讲解时应突出推荐系统升级的必要性：用户面对大量信息时筛选成本高，传统算法依赖人工特征工程，难以适应兴趣变化；新用户和新内容缺乏历史数据，推荐容易失准；数据分散在不同业务线，用户画像不完整；黑盒推荐缺乏解释，降低用户信任。

## 第 7 页：项目目标：构建智能协同的个性化推荐平台

本页从“核心解决方案架构”和“关键功能落地实现”两方面描述目标。核心能力包括多 Agent 协同决策、多模态内容理解、RAG 增强推荐解释、推荐全链路自动化闭环；落地功能包括实时响应推荐流、动态自适应用户画像、结构化推荐理由展示和运营效果可视化看板。

讲解重点是项目目标不只是“给出推荐结果”，而是构建推荐闭环：内容理解、用户画像、推荐决策、解释生成、反馈采集和运营分析共同工作，实现千人千面和持续优化。

## 第 8 页：系统架构设计章节页

本页进入第三章，强调系统采用高可用、可扩展、易维护的分布式架构，通过模块化和分层设计实现业务解耦与资源调度。

讲解时可作为架构部分过渡，说明后续将从整体架构、代码管理、功能矩阵等角度介绍系统如何支撑智能推荐业务。

## 第 9 页：整体架构

本页展示系统四层架构：前端交互层、后端服务层、AI 智能引擎层和数据存储层。前端基于 Vue 3、Element Plus、ECharts，通过 SSE 和 REST API 与后端通信；后端基于 FastAPI、SQLAlchemy 2.0、Pydantic、JWT；AI 层集成 YOLOv11、通义千问和 pgvector；数据层使用 PostgreSQL、Redis、MinIO。

讲解重点是分层解耦：前端负责交互与可视化，后端负责业务接口和权限，AI 层负责检测、语义理解和向量检索，数据层负责结构化数据、缓存和对象存储。这种架构便于扩展、维护和故障隔离。

## 第 10 页：中心管理与 API 接口

本页分为代码管理与协作规范、核心业务 API 接口概览。代码管理部分强调 GitHub 私有仓库、分支管理、版本追溯、Code Review 和“提交-评审-合并”闭环。

讲解时可从工程管理角度说明：推荐系统不仅依赖模型效果，也依赖稳定的工程流程。通过版本控制、分支隔离和代码评审，可以降低多人协作中的冲突风险，提高系统可维护性。

## 第 11 页：项目核心功能与技术特点

本页将系统能力整理成核心功能矩阵和技术架构优势。功能包括智能推荐流、全维 AI 分析、结构化内容库、角色化业务流转、自动生成报告；技术特点包括 YOLOv11 精准识别、记忆式智能分析、多维度数据看板、轻量级快速部署和 RBAC 分级权限管控。

讲解重点是系统兼顾 AI 能力和业务闭环：既有目标检测和 LLM 分析，也有内容库、报告、权限、部署和看板，能支撑完整的推荐业务流程。

## 第 12 页：核心算法实现章节页

本页进入第四章，说明将拆解核心算法的设计思路、参数调优策略和实际性能表现。页面强调算法要为系统提供高精度、低延迟的智能决策支撑。

讲解时可作为算法部分引入，说明算法并不是孤立训练模型，而是要能接入后端、支撑实时推荐，并在准确率和响应速度之间取得平衡。

## 第 13 页：模型介绍：双模型协同工作

本页介绍双模型协同：YOLOv11 用于内容理解与精准特征定位，通义千问 Qwen-plus 用于综合研判与智能决策分析。YOLO 输出检测框坐标、置信度和类别标签；LLM 融合检测结果与用户历史行为，生成结构化推荐理由和个性化建议。

讲解重点是“检测模型负责看见，语言模型负责理解和表达”。YOLO 将非结构化内容转为结构化特征，LLM 再结合用户画像、历史行为和推荐目标进行分析，使推荐结果更可解释、更贴近用户需求。

## 第 14 页：数据集与图像处理

本页介绍数据集 MultiModal-Rec10，包含文本、图片、视频等 10 万条多模态内容数据，并由 3 位数据分析师标注，覆盖新闻、娱乐、科技、教育等 10 类标签。页面还介绍数据增强与预处理体系，包括空间几何变换、色彩与光照扰动、标准化预处理、高级特征增强。

讲解重点是数据质量决定模型上限。通过裁剪、缩放、翻转、亮度对比度调整、归一化、CLAHE、中值滤波和模糊操作，可以提升模型对多样内容环境的适应能力，增强泛化和鲁棒性。

## 第 15 页：模型训练与效果

本页介绍训练策略与评估结果。检测模型采用 YOLOv11n 轻量级版本，在 640×640 分辨率下训练 30 个 epoch，优化小目标特征提取和定位精度。LLM 侧通过 Prompt Engineering 和领域数据增强复杂推荐场景理解。效果指标包括检测模型 mAP@50 为 89.6%，LLM 综合分析准确率为 92.3%。

讲解时应突出“轻量化”和“可靠性”：YOLOv11n 能兼顾推理速度和精度，适合在线系统；LLM 通过提示工程和领域知识增强，让推荐解释更符合业务语境。

## 第 16 页：前端功能模块与开发章节页

本页进入第五章，概括前端建设方向：交互与体验设计、组件化开发体系、性能优化与适配。

讲解时可说明前端目标是降低用户使用门槛，让用户能够自然上传内容、查看推荐结果、理解推荐理由，并让运营或管理人员通过数据看板掌握系统效果。

## 第 17 页：前端技术选型与架构

本页列出前端技术栈：Vue 3、Element Plus、Vite、Pinia、Vue Router 4、ECharts。同时说明架构规范：组件化拆分、路由守卫和动态鉴权、Axios 统一封装、ESLint 与自动化部署。

讲解重点是前端工程化价值：Vue 3 提供响应式开发基础，Element Plus 提供稳定组件，Vite 提高开发效率，Pinia 管理全局状态，ECharts 支撑统计可视化，路由守卫和鉴权保障不同角色看到不同功能。

## 第 18 页：前端核心功能实现

本页将前端能力分为智能多模态输入、AI 推理全链路闭环、可视化与数据洞察。具体包括单图即时检测、交互触发与校验、模型推理与计算、结果解析与呈现、批量异步处理、ZIP 自动解析、Canvas 高性能渲染、ECharts 数据看板。

讲解重点是前端不仅是展示页面，而是参与完整业务链路：上传前进行格式、大小、分辨率校验；上传后调用 AI 接口；收到 JSON 结果后渲染标注框、热力图和统计图；批量任务通过进度条反馈处理状态。

## 第 19 页：后端架构设计章节页

本页进入第六章，聚焦高可用、高并发的后端架构设计，强调数据库选型、索引优化、数据持久化和安全稳定的底层支撑。

讲解时可说明后端是系统核心枢纽：它连接前端请求、AI 模型、知识库、数据库、缓存和对象存储，负责把推荐业务从一次模型调用变成可管理、可追溯、可扩展的服务体系。

## 第 20 页：技术栈与架构设计

本页从四个方面介绍后端架构：核心技术栈、分层架构体系、智能与检索引擎、高可用存储与容灾。核心技术包括 FastAPI、LangGraph、通义千问、GPT-4o-mini、JWT、YOLOv11、text-embedding-v3、pgvector、PostgreSQL、MinIO、Redis。

讲解重点是后端采用现代化全栈和分层解耦架构：FastAPI 处理高并发接口，LangGraph 编排多 Agent，JWT 做安全认证，RAG 和 pgvector 增强事实检索，Redis 与内存降级保障高可用，MinIO 和 PostgreSQL 分别管理文件与业务数据。

## 第 21 页：多 Agent 协作系统 · 整体拓扑

本页说明为什么需要多 Agent：推荐任务涉及内容理解、用户画像、策略生成等多维度需求，单一 Agent 难以兼顾。Supervisor 统一接收请求、拆解任务、分发给专家 Agent，并汇总输出。

讲解重点是多 Agent 架构的价值：专家 Agent 各司其职，提升复杂任务处理深度；Supervisor 维护统一交互出口，让用户感觉仍然是在和一个智能系统对话；任务从需求识别到推荐、解释和反馈可以自动闭环流转。

## 第 22 页：Supervisor 总管 — 两级智能路由

本页介绍两级智能路由机制。第一级是规则快速路由，基于关键词精准匹配，高频意图可毫秒级分发；第二级是 LLM 语义深度路由，针对复杂模糊请求，读取最近 16 轮上下文，判断真实意图并匹配专家 Agent。最后由 Supervisor 统一编排输出。

讲解重点是规则路由与 LLM 路由的互补：规则路由速度快、成本低，适合常见明确请求；LLM 路由理解能力强，适合模糊表达和跨轮对话；统一输出则避免多个 Agent 回复风格不一致。

## 第 23 页：五大专业 Agent 各司其职

本页介绍专业 Agent 分工，包括内容分析 Agent、智能推荐 Agent、推荐解释 Agent、智能问答 Agent、用户画像 Agent，以及协同调度与闭环优化。页面强调在推荐和用户画像执行前，系统会自动从数据库加载用户档案、近期偏好和历史交互记录。

讲解重点是推荐系统要“先理解用户，再推荐内容”。内容分析 Agent 把素材转为标签和特征；推荐 Agent 融合内容、画像和知识库生成列表；解释 Agent 说明推荐依据；QA Agent 基于知识库答疑；用户画像 Agent 根据点击、停留、互动持续更新兴趣权重；调度机制收集反馈并优化策略。

## 第 24 页：知识库 + RAG：检索增强生成技术

本页解释 RAG 的原理：LLM 在回答前先从外部知识库检索相关资料作为参考，而不是只依赖模型记忆。页面强调 RAG 的价值是提升专业性和可信度、减少幻觉、支持知识更新和来源追溯。

讲解重点是 RAG 解决大模型知识滞后和领域知识不足的问题。标准流程是用户提问、问题向量化、向量库匹配、筛选 Top-N 文档片段、注入 Prompt、LLM 生成带依据的回答。

## 第 25 页：知识库构建流程与管理

本页介绍知识库流水线：将 Markdown 文档按“段落 + 标题”智能分块，每块约 500 字、重叠 50 字；调用 Embedding API 生成 1024 维向量；存入 PostgreSQL + pgvector，并建立 IVFFlat 索引。检索时使用向量余弦相似度，并设置 0.35 阈值过滤低相关结果。

讲解重点是知识库不仅要能检索，还要能管理和容错。系统保留文档元数据以支持溯源；向量索引支撑高并发检索；当 pgvector 不可用时自动切换到内存向量存储；知识库未初始化时隔离 RAG 模块，避免影响核心业务。

## 第 26 页：三层记忆架构

本页介绍运行时 State、Redis 缓存层、PG 持久化存储、智能上下文融合四部分。State 作为单次请求内的热存储，请求结束释放；Redis 按“用户 + 会话”缓存近期对话，TTL 为 1 小时；PostgreSQL 永久保存会话和消息；上下文融合自动注入身份、病例、检测记录或历史交互信息。

讲解重点是多轮对话需要记忆能力。三层设计兼顾实时性和可追溯性：State 保证单轮 Agent 协同，Redis 提供高频读取，PostgreSQL 保存完整历史，智能上下文融合让系统能跨轮次恢复用户语境。

## 第 27 页：后端全景 — API 模块与数据库

本页总结后端全景：采用微服务分层设计和混合存储集群，通过 API 网关统一路由与鉴权。底层使用业务数据库、Redis 缓存和 pgvector 向量检索。页面列出 6 类模块：智能交互与知识库、推荐与内容服务、用户与权限治理、模型与系统管控、数据运营与审计、高可用架构原则。

讲解重点是后端模块化设计：聊天和知识库负责 AI 交互，推荐服务生成推荐和报告，用户权限模块负责 JWT 与 RBAC，模型模块管理训练和部署，运营模块提供指标和审计，高可用原则通过服务降级、资源复用、异常捕获和密码加密保障稳定安全。

## 第 28 页：项目成果展示章节页

本页进入第七章，强调项目在效率提升、成本优化和用户体验方面取得成果，并为后续规模化扩张奠定基础。

讲解时可作为成果展示过渡，说明后续将把前面介绍的架构、算法和工程能力落到实际系统功能上。

## 第 29 页：项目成果展示

本页展示最终系统能力，包括分级权限认证、智能影像工作站、医疗大模型交互、全周期病例管理、多维统计分析、检测记录回溯。页面强调系统采用医疗级现代化 UI，融合 AI 辅助诊断、数据可视化与病例管理，降低医护人员学习成本。

讲解重点是成果已经覆盖完整应用场景：管理员、医生、患者三类角色隔离；支持单图、批量和压缩包上传；大模型支持流式问答和健康建议；医生维护病例，患者查看报告；看板展示检测总量和趋势；历史记录支持多条件筛选和导出。

## 整体总结

这份 PPT 的整体逻辑是完整的项目汇报结构：先介绍团队分工，再说明传统推荐系统面临的信息过载、冷启动、数据孤岛和可解释性不足等痛点，随后提出多 Agent、RAG、多模态内容理解和动态用户画像共同支撑的智能推荐平台目标。

从技术实现看，系统采用前端、后端、AI 引擎和数据存储的分层架构。前端负责交互、上传、可视化和角色化页面；后端以 FastAPI 为核心，承担鉴权、业务 API、多 Agent 编排、模型调用、知识库检索和数据持久化；AI 层通过 YOLOv11 进行特征识别，通过 LLM 进行语义分析和推荐解释；数据层通过 PostgreSQL、pgvector、Redis、MinIO 等组件完成结构化数据、向量、缓存和文件管理。

后端部分是整套系统的核心亮点。Supervisor 两级路由将规则快速匹配和 LLM 语义判断结合起来，在效率和理解能力之间取得平衡；专业 Agent 将内容分析、推荐、解释、问答和用户画像拆分处理，提高了复杂任务的可维护性；RAG 知识库为推荐理由和问答提供可追溯依据；三层记忆架构让系统具备连续上下文能力。

整体来看，该方案希望把传统“黑盒推荐”升级为“可解释、可交互、可追溯、可迭代”的智能推荐平台。它的价值不只在于生成推荐结果，更在于把内容理解、用户画像、推荐决策、解释生成、反馈闭环和运营分析连接起来，为个性化服务和持续优化提供完整技术基础。
